## Context

Roger é um engenheiro reverso e expert em infraestrutura. Ele enfrentou um problema crítico de rate limit no opencode que impedia o OpenClaw de funcionar.

### O Problema
- O modelo principal configurado era opencode/big-pickle
- Toda vez que o OpenClaw tentava usar a API do opencode, caía em rate limit
- Muitas sessões falharam com "[assistant turn failed]" ao longo de 20/05

### A Solução (descoberta por Roger)
1. Ele percebeu que **rodando o Big Pickle diretamente pelo opencode** (interface original) **NÃO** caía em rate limit
2. Como ambos usavam a mesma API key, essa diferença não fazia sentido — então foi investigar
3. **Inspecionou via proxy** as requisições HTTP feitas pelo opencode original vs as feitas pelo OpenClaw
4. Encontrou diferenças nos **atributos do JSON de requisição** — a estrutura/formatação era diferente
5. **Forçou o OpenClaw a montar a requisição no mesmo formato do opencode**, efetivamente se passando pelo opencode para o servidor backend
6. Resultado: Big Pickle rodando sem rate limits no OpenClaw agora

### Lições
- O rate limit não era baseado em API key, mas em **assinatura/formato da requisição**
- Proxy inspection foi a chave pra achar a diferença
- Roger é bom pra caralho em engenharia reversa 🎯
